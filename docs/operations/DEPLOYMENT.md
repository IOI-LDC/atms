# ATMS Deployment Guide

## Prerequisites

- Docker Engine 24+
- Docker Compose v2.20+
- 2 GB RAM minimum, 4 GB recommended
- 10 GB disk for images, additional for attachments and database

## Environment Setup

1. Copy the example environment file:

```sh
cp .env.example .env
```

2. Generate an application key:

```sh
docker compose run --rm api php artisan key:generate --env
```

3. Edit `.env` and set:

| Variable | Required | Description |
|----------|----------|-------------|
| `APP_KEY` | Yes | Generated by `key:generate` |
| `APP_URL` | Yes | Public URL (e.g., `https://atms.example.com`) |
| `DB_PASSWORD` | Yes | PostgreSQL password |
| `LDC_ERP_CLIENT_ID` / `LDC_ERP_CLIENT_SECRET` | For ERP sync | OAuth2 client credentials for LDC ERP token acquisition |
| `LDC_ERP_BASE_URL` | For ERP sync | Base URL of the LDC ERP API |
| `GRAPH_TENANT_ID` | For email | Azure AD tenant (notification app — separate Entra app from `LDC_ERP_*`) |
| `GRAPH_CLIENT_ID` | For email | Azure AD app (client) ID |
| `GRAPH_CLIENT_SECRET` | For email | Azure AD client secret (certificate is the stronger alternative) |
| `GRAPH_MAILBOX` | For email | Sender mailbox (e.g. `notification@ldc.com.ly`) |

## Build and Start

```sh
docker compose build
docker compose -f compose.yaml -f compose.production.yaml up -d
```

## Initialize Database

```sh
docker compose exec api php artisan migrate --force
docker compose exec api php artisan db:seed --force
```

## Create First Administrator

```sh
docker compose exec api php artisan atms:make-admin \
    --name="Admin" \
    --email=admin@example.com \
    --password=changeme
```

## Verification

```sh
docker compose exec api php artisan test
sh scripts/test-integration.sh
sh scripts/security-smoke.sh
```

## Production Configuration

`compose.production.yaml` adds `restart: unless-stopped` to all services. The `compose.override.yaml` (development only) is automatically merged by Docker Compose but should not be present in production.

To deploy without the override:

```sh
docker compose -f compose.yaml -f compose.production.yaml up -d
```

### Architecture

- **nginx**: Public-facing reverse proxy. Serves HTTPS on port 443 in production; HTTP on port 80 in development only
- **api**: PHP-FPM application (port 9000 internal only)
- **postgres**: PostgreSQL 17 database
- **queue**: Queue worker (`queue:work --tries=3`)
- **scheduler**: Scheduler (`schedule:work`)

### Volumes

- `pgdata`: PostgreSQL data (persistent)
- `attachments`: Uploaded files (persistent)

## TLS / HTTPS Setup (Production)

The API must serve all traffic over HTTPS in production. Development uses HTTP
on port 80; production must terminate TLS at nginx.

### Certificate

Use Let's Encrypt with certbot for a free, auto-renewing certificate:

```sh
# Install certbot on the VPS host (not inside a container)
sudo apt install certbot

# Obtain certificate via standalone or webroot mode
sudo certbot certonly --standalone -d atms.example.com
```

Alternatively, if a cloud load balancer or CDN sits in front of the VPS,
terminate TLS there and forward to nginx on an internal port.

### nginx TLS Configuration

Mount the certificate and key into the nginx container and configure:

```nginx
server {
    listen 443 ssl;
    server_name atms.example.com;

    ssl_certificate     /etc/nginx/certs/fullchain.pem;
    ssl_certificate_key /etc/nginx/certs/privkey.pem;

    # ... proxy_pass to api:9000 ...
}

server {
    listen 80;
    server_name atms.example.com;
    return 301 https://$host$request_uri;
}
```

### Certificate Renewal

Certbot certificates expire every 90 days. Automate renewal:

```sh
# Add to crontab on the VPS host
0 3 * * * certbot renew --quiet --post-hook "docker compose restart nginx"
```

## Strong Secrets

All production secrets must be cryptographically strong and unique. No default
or placeholder values are permitted.

| Secret | Minimum Length | Generated By |
|---|---|---|
| `APP_KEY` | 32 chars (base64) | `php artisan key:generate` |
| `DB_PASSWORD` | 32 chars | `openssl rand -base64 32` |
| API client secrets | 64 chars | Application (per remote system) |
| Admin initial password | 12 chars | Set at deploy; changed on first login |

Never commit a `.env` file containing real secrets. The `.env.example` is the
committed template with placeholder values.

## Port Lockdown

Only the HTTPS port (443) may be exposed to the public internet. Enforce this
at both Docker and host firewall levels:

```sh
# Host firewall (ufw example)
sudo ufw default deny incoming
sudo ufw allow 443/tcp
sudo ufw allow 80/tcp     # HTTP redirect only
sudo ufw allow ssh
sudo ufw enable
```

PostgreSQL (5432) and PHP-FPM (9000) must never be bound to `0.0.0.0`.
Verify Docker port mappings in `compose.yaml` and `compose.production.yaml`
bind these services to `127.0.0.1` or leave them on the internal Docker network.

See [../03-backend/SECURE_REMOTE_API_ACCESS.md](../03-backend/SECURE_REMOTE_API_ACCESS.md)
for the full production security specification and deployment checklist.

## Scheduled Tasks

| Schedule | Task | Description |
|----------|------|-------------|
| Monday 02:00 | `erp-asset-sync` | Sync assets from ERP |
| Monday 03:00 | `erp-part-sync` | Sync parts from ERP |
| Daily 06:00 | `pm-evaluation` | Evaluate PM rules and generate work orders |

## Monitoring

- **Liveness**: `GET /api/health/live` — returns 200 if PHP-FPM is running
- **Readiness**: `GET /api/health/ready` — checks database and storage connectivity
- **Queue**: Monitor `jobs` table for failed jobs

## SharePoint Integration

ATMS is hosted as a standalone web application. SharePoint contains only a normal link to the ATMS URL. Users authenticate through ATMS itself using Sanctum SPA cookie authentication. Employee directory data is imported into the local employee directory through the configured SharePoint integration when available; Power Automate is not used by ATMS.
