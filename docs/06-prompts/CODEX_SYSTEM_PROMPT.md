# Codex System Prompt

You are assisting with implementation of ATMS, the Asset Maintenance Tracking System.

## Product Summary

ATMS is a simplified operational maintenance application. It reads fixed assets
and parts from ERP, keeps a local operational copy, tracks asset usage/location,
manages Maintenance Requests, converts approved requests into Work Orders, and
provides maintenance history derived from authoritative operational records.

The core workflow is:

**Maintenance Request → Maintenance Manager Approval → Work Order → Closure → Asset Maintenance History**

## Locked Stack

- Backend: Laravel 13 API backend
- PHP runtime: PHP 8.4
- Database: PostgreSQL
- Frontend: Vue 3 + TypeScript + Tailwind
- Deployment: one Docker Compose service model for local OrbStack development and VPS production, with environment-specific overrides
- Background jobs: Laravel Queues using PostgreSQL database queues for MVP
- Scheduled jobs: Laravel Scheduler
- Attachments: Laravel local storage on a persistent Docker volume
- Mock ERP: minimal Laravel 13/PHP 8.4 service using disposable seeded SQLite, in a separate profile-based container, internal Docker network only, with a read-only asset/part API
- Mock ERP authentication: static service API key in an environment-configured HTTP header
- Authentication: Laravel Sanctum SPA cookie/session authentication; bearer API tokens are not the primary authentication method
- Redis and MinIO: optional future upgrades, excluded from the default MVP deployment

## Important Product Rules

1. ERP remains the source of truth for fixed assets and parts.
2. ATMS must not handle capitalization, disposal, depreciation, procurement, warehouse accounting, labor tracking, or financial asset tracking.
3. Work Orders must not be created directly by normal users.
4. A Work Order is created only when a Maintenance Request is approved by a Maintenance Manager.
5. Preventive Maintenance Requests are generated by the system from PM Rules.
6. Corrective Maintenance Requests are created by users.
7. Parts can be recorded as used on Work Orders, but no parts costing or stock valuation is included.
8. Attachments are required for Assets and Parts, and should also support Maintenance Requests and Work Orders.
9. Each user has exactly one of six seeded fixed roles: Administrator, Maintenance Manager, Technician, Logistics, Requester, or Viewer. Do not add multiple roles, custom roles, or granular permission management.
10. Logistics may update asset physical location but has no maintenance approval, Work Order execution, PM Rule, or administration permissions.
11. Do not implement technician hour logs, labor rates, labor costs, timesheets, or productivity reporting.
12. PM Rules apply only to individual ERP-linked assets; grouped or category-level rules are out of scope.
13. Work Order parts are operational usage records only; do not implement stock, warehouse, procurement, valuation, or costing behavior.
14. Logistics exists only for physical location updates/history and does not introduce logistics documents or workflows.

## Coding Guidance

- Keep controllers thin.
- Use Form Requests for validation.
- Use Resource classes for API responses.
- Put workflow transitions in Action classes.
- Use policies or permission checks for authorization.
- Add tests for important workflow transitions.
- Avoid adding out-of-scope functionality unless explicitly requested.
