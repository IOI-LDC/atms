# Secure Remote API Access

**Status:** Implemented in code (Phase 1). Token endpoint, client management, and ability enforcement are live.

## Context

A remote system, hosted on the other side of the world, must securely read data
from the ATMS/SM/AM backend API. The backend is deployed on a VPS through Docker.

Concrete use case: an external system (VJ) consumes Parts QTY data from the SM
(Store Management) subsystem tables via the API. The external system only
requires **read** access — it never creates, updates, or deletes data.

Currently the API is served over plain HTTP with only browser-session
(Sanctum SPA cookie) authentication. This is insufficient for secure
machine-to-machine access across the public internet.

## Three Non-Negotiable Production Requirements

Before any remote system can connect to the API in production, the following
three requirements must be satisfied:

---

### 1. HTTPS (TLS) via Reverse Proxy

The API is currently served over **HTTP only** — nginx listens on port 80 with
no TLS termination.

**Production MUST have TLS encryption on all API traffic.** The recommended
approach:

- Terminate TLS at the nginx reverse proxy (the public-facing container).
- Configure nginx to listen on **port 443** with a valid TLS certificate.
- Redirect all HTTP (port 80) traffic to HTTPS (port 443).
- Use **Let's Encrypt** (certbot) with auto-renewal, or a cloud-provider TLS
  certificate if a load balancer sits in front of the VPS.
- The internal Docker network (nginx → PHP-FPM on port 9000) remains
  unencrypted — this is acceptable because it never leaves the host.

**Why:** Without TLS, API tokens, credentials, and all data traverse the public
internet in plaintext. A remote system on the other side of the world cannot
securely authenticate or read data over an unencrypted channel.

---

### 2. API Bearer Tokens for Machine-to-Machine Authentication

The backend currently uses only **Sanctum SPA cookie/session authentication** —
designed for browser-based human users. There is no token endpoint for
non-browser clients.

**Production MUST provide a bearer-token authentication mechanism for
machine-to-machine access.** This is separate from web-user auth (which
continues to use SPA cookies as documented in `ARCHITECTURE.md`).

Requirements for the token system:

| Requirement | Detail |
|---|---|
| **Auth mechanism** | Laravel Sanctum API tokens (`HasApiTokens` trait) — already included in Laravel Sanctum, separate from SPA cookie auth |
| **Token endpoint** | A new `POST /api/auth/token` endpoint that accepts client credentials and returns a Bearer token |
| **Read-only scope** | Tokens must be scoped to read-only access. The `abilities` column on `personal_access_tokens` must restrict the token to `read` only |
| **Token lifecycle** | Long-lived tokens (suitable for automated systems). Tokens must be revocable by Administrators |
| **Token management** | Administrators can create, list, and revoke API tokens through an admin interface or CLI command |
| **Audit** | Token creation, usage (last used timestamp), and revocation must be logged to the audit trail |

**What this is NOT:**

- This is **not** replacing SPA cookie auth for browser users. The rule in
  `ARCHITECTURE.md` — *"Primary web authentication must not use bearer API
  tokens"* — still applies. Web users authenticate via session cookies.
- This is **not** OAuth2/OpenID Connect. A simple client-credentials pattern
  using Sanctum's built-in token support is sufficient for MVP.

**Minimum viable token endpoint contract:**

```
POST /api/auth/token
Content-Type: application/json

{
  "client_id": "string",
  "client_secret": "string"
}

Response 200:
{
  "token": "1|abc123...",
  "abilities": ["read"],
  "expires_at": null
}
```

Client credentials (`client_id` / `client_secret`) are stored as an
Administrator-managed API client record in the database, not as `.env`
variables. Each remote system gets its own credential pair.

**Read-only enforcement:** Every endpoint served to a bearer-token-authenticated
request must enforce the `read` ability. Write operations (POST/PATCH/PUT/DELETE)
must return `403 Forbidden` when authenticated via a read-only token, regardless
of the user role associated with the token.

---

### 3. Strong Secrets and Locked-Down Ports

Production must not use default, weak, or publicly known secrets. Only the
minimum necessary ports may be exposed to the internet.

#### 3a. Strong Secrets

Every secret in production must be a unique, cryptographically strong value:

| Secret | Requirement |
|---|---|
| `APP_KEY` | Generated by `php artisan key:generate` — never hardcoded or copied from another environment |
| `DB_PASSWORD` | Minimum 32 random characters — never `secret`, `password`, `postgres`, or any default |
| `LDC_ERP_CLIENT_SECRET` | Provided by the client's ERP team — never stored in source control |
| `GRAPH_CLIENT_SECRET` | Provided by the client's Azure AD admin — never stored in source control |
| API client secrets | Generated per remote system by the application — minimum 64 random characters |
| `atms:make-admin --password` | A strong initial password set at deploy time, changed on first login |

No `.env` file containing real secrets may be committed to source control.
The `.env.example` file must contain only placeholder values.

#### 3b. Locked-Down Ports

Only the **HTTPS port (443)** may be exposed to the public internet. All other
ports must be bound to `localhost` or the internal Docker network only:

| Port | Service | Exposure |
|---|---|---|
| 443 | nginx (HTTPS) | **Public** — the only internet-facing port |
| 80 | nginx (HTTP redirect) | Public — redirect only; no application data served |
| 9000 | PHP-FPM | Internal Docker network only |
| 5432 | PostgreSQL | Internal Docker network only — never bound to `0.0.0.0` |

**Host firewall:** Use `ufw` or the VPS provider's cloud firewall to enforce
port restrictions at the host level as a defense-in-depth measure, independent
of Docker's port mappings.

**nginx configuration must:**
- Listen on 443 with TLS.
- Reject any request that does not present a valid `Authorization: Bearer <token>`
  header on API routes that require it.
- Return standard HTTP error responses — never expose stack traces, framework
  details, or internal IP addresses.

---

## Deployment Checklist (Production Gate)

Before the first remote system connects to production:

- [ ] TLS certificate installed and auto-renewing on nginx (port 443).
- [ ] HTTP (port 80) redirects to HTTPS.
- [ ] All non-HTTPS ports firewalled at host level.
- [ ] All `.env` secrets replaced with strong, unique values.
- [ ] No `.env` file committed to source control.
- [ ] Bearer token endpoint implemented and tested.
- [ ] Token-scoped read-only enforcement verified on all API routes.
- [ ] API client credentials generated per remote system.
- [ ] Audit logging confirmed for token operations.
- [ ] `APP_URL` set to `https://...` in production `.env`.
- [ ] `APP_DEBUG=false` and `APP_ENV=production`.
- [ ] Sanitized `APP_URL` set in nginx config; internal error pages hide stack traces.
