# PM Rules Implementation Plan

**Goal:** Build a full PM Rules feature with its own sidebar item (Admin + Manager), supporting per-asset preventive maintenance schedules with date/reading/combined triggers, multi-level maintenance (L1-L4), human-readable schedule display, next-due estimates with status indicators, cumulative level reset, and a rich detail page with MR history.

**Design Rule:** Follow existing page conventions. Reference pages are the source of truth for layout, styling, and interaction patterns. The implementer has freedom on component decomposition, field arrangement, and internal naming — but must match the look and feel of existing pages.

## Backend Prerequisites (7 — flagged for backend team)

These must be completed before or alongside the frontend. Each is a backend-only change.

### 1. PmRulePolicy: restrict configuration to Admin only

File: `backend/app/Policies/PmRulePolicy.php`

Currently `create`, `update`, `deactivate`, `reactivate` allow Admin AND Manager. Change to **Admin only**:
- `viewAny` / `view`: Admin + Manager (keep)
- `evaluate`: Admin + Manager (keep)
- `create` / `update` / `deactivate` / `reactivate`: **Admin only**

This enforces "Admin configures, Manager views only."

### 2. PmRuleResource: add usage_reading_type

File: `backend/app/Http/Resources/PmRuleResource.php`

The resource does NOT currently include the reading type. Add:
- `usage_reading_type` — whenLoaded: `{ id, name, unit }`

The controller `show` method already eager-loads `usageReadingType`. The `index` query must also eager-load it.

### 3. PmRuleResource: add suppressions

Same file. Add:
- `suppressions` — whenLoaded: array of `{ id, suppressed_until_date, suppressed_until_reading, source_mr_id }`

The controller `show` method already eager-loads `suppressions`.

### 4. PmRuleResource: add computed next-due + status

Same file. Add computed fields:
- `next_due_date` — `last_triggered_date + interval_days` (null if not date-based)
- `next_due_reading` — `last_triggered_reading + interval_reading` (null if not reading-based)
- `pm_status` — `"ok"` / `"soon"` / `"due"` (computed from progress; needs asset's current confirmed meter reading for reading-based rules)
- `progress_percentage` — 0-100 (date progress or reading progress, whichever is higher for combined rules)

Status thresholds: `ok` < 60%, `soon` 60-80%, `due` >= 80% (or past threshold).

For reading-based progress, the resource needs the asset's current confirmed meter reading. Load it via the asset relation.

### 5. pm_rules table: add maintenance_level column

Migration: add nullable string column `maintenance_level` to `pm_rules`. Default null. Values: `"L1"`, `"L2"`, `"L3"`, `"L4"`, or custom free-text, or null.

Add to PmRule model `$fillable` and to the resource output.

Add validation to `PmRuleController::store`: `'maintenance_level' => ['nullable', 'string', 'max:10']`.

### 6. MaintenanceRequest index: add pm_rule_id filter

File: `backend/app/Queries/MaintenanceRequests/MaintenanceRequestIndexQuery.php` (or equivalent)

Add `pm_rule_id` as a supported filter on `GET /api/maintenance-requests`. This lets the PM Rule detail page fetch MRs generated by a specific rule: `GET /api/maintenance-requests?pm_rule_id={id}`.

### 7. WO closure: cumulative reset of lower-level baselines

File: `backend/app/Actions/WorkOrders/CloseWorkOrder.php` (or `CompleteWorkOrder.php`)

When a WO closes that originated from a PM-generated MR:
1. Find the PM rule via `maintenanceRequest.pm_rule_id`
2. If the rule has a `maintenance_level` matching `L[1-4]` pattern, parse the numeric suffix (e.g., "L3" → 3)
3. Find all other active PM rules on the same asset with a LOWER numeric level (e.g., L1, L2 if closing L3)
4. Reset their `last_triggered_date` to now and `last_triggered_reading` to the asset's current confirmed reading
5. Audit-log the reset

This only applies to levels with numeric suffixes (L1-L4). Custom free-text levels are independent — not affected by cumulative reset.

## Schedule Display

A frontend helper function composes a human-readable string from the rule + reading type. Examples:

| Trigger type | Display |
|---|---|
| `date` (interval_days=90) | "Every 3 Months" |
| `reading` (interval_reading=500, type="Operating Hours") | "Every 500 Operating Hours" |
| `date_or_reading` (180 days, 500 hrs) | "Every 500 Operating Hours or 6 Months, whichever comes first" |

Day-to-period conversion logic:
- Divisible by 365 → "X Years"
- Divisible by 30 → "X Months"
- Divisible by 7 → "X Weeks"
- Otherwise → "X Days"

The reading unit comes from `usage_reading_type.name` (e.g., "Operating Hours" → display "Operating Hours", abbreviated to "Ops Hrs" where space-constrained).

## Multi-Level Setup Form

The creation form (`PmRuleForm` side sheet) has a mode toggle:

**Single Rule mode:**
- Asset selector (reuse `AssetCombobox`)
- Maintenance Level (optional: L1/L2/L3/L4/None/Custom dropdown)
- Rule Name
- Trigger Type (Calendar / Usage / Whichever comes first)
- Conditional fields based on trigger type
- Description (optional)

**Multi-Level Setup mode:**
- Asset selector (single — applies to all levels)
- Three rows (L1, L2, L3) — each with:
  - Rule Name (pre-filled with level label, editable)
  - Trigger Type (defaults to same as L1, individually changeable)
  - Interval inputs (days + reading)
- Submit creates all rules via sequential `POST /api/pm-rules` calls
- Per-row error handling — if L2 fails, L1 and L3 still succeed, L2 shows error inline

## Rule Detail Page

Route: `/pm-rules/:ruleId`

Follows `AssetDetailView.vue` pattern — back button, header with status/level/trigger badges, then data-card sections.

**Sections:**
1. **Schedule (hero card)** — the human-readable schedule string prominently displayed, with raw values below (trigger type, interval days, interval reading, reading type)
2. **Asset** — asset tag + name, clickable link to `/assets/{id}`
3. **Next-Due & Baselines** — `next_due_date` / `next_due_reading` with 🟢🟡🔴 status indicator + progress. Last triggered date/reading. Created by + date.
4. **Suppressions** — active suppression windows (suppressed_until_date, suppressed_until_reading, originating MR link). Shown only if suppressions exist.
5. **MR History** — table of maintenance requests generated by this rule (`GET /api/maintenance-requests?pm_rule_id={id}`). Columns: MR number (link), status badge, generated date, priority.

**Action bar:**
- **Edit** (Admin only) — opens edit side sheet
- **Evaluate Now** (Admin + Manager) — calls `POST /api/pm-rules/{id}/evaluate`, shows result inline or toast
- **Deactivate / Reactivate** (Admin only) — confirm dialog. Deactivation blocked (409) if active MR/WO chain exists — show the blocking MR with link.

## API Contracts

### PM Rules
```
GET /pm-rules?sort=...&is_active=...     → cursor-paginated { data: PmRule[], links, meta }
GET /pm-rules/{rule}                      → { data: PmRule }

PmRule: {
  id, name, description, trigger_type, is_active,
  maintenance_level,                      ← NEW (prerequisite #5)
  interval_days, interval_reading,
  last_triggered_date, last_triggered_reading,
  next_due_date, next_due_reading,        ← NEW (prerequisite #4)
  pm_status, progress_percentage,         ← NEW (prerequisite #4)
  asset: { id, name, erp_asset_code },
  usage_reading_type: { id, name, unit }, ← NEW (prerequisite #2)
  suppressions: [ { id, suppressed_until_date, suppressed_until_reading, source_mr_id } ],  ← NEW (#3)
  created_by: { id, name },               ← Admin/Manager only
  created_at
}

POST /pm-rules
  Body: { asset_id, name, description?, trigger_type, interval_days?, interval_reading?,
          usage_reading_type_id?, maintenance_level? }   ← level is NEW
  → 201 { data: PmRule }
  Auth: Admin only (prerequisite #1)

PATCH /pm-rules/{rule}
  Body: { name?, description?, interval_days?, interval_reading?, maintenance_level? }
  → 200 { data: PmRule }
  Auth: Admin only (prerequisite #1)

POST /pm-rules/{rule}/deactivate     → 200 or 409 (active chain)   Auth: Admin only
POST /pm-rules/{rule}/reactivate     → 200                         Auth: Admin only
POST /pm-rules/{rule}/evaluate       → 200 "not due" or 201 { data: MR }   Auth: Admin + Manager
POST /pm-rules/evaluate              → 200 "Evaluated N rules, generated M requests"   Auth: Admin + Manager
```

### MR History (for detail page)
```
GET /maintenance-requests?pm_rule_id={id}   ← NEW filter (prerequisite #6)
  → cursor-paginated MR list, same shape as existing MR endpoint
```

## Risk Notes

1. **Cumulative reset complexity.** The WO closure action must correctly identify the originating PM rule (via MR → pm_rule_id), parse the level, and reset lower-level rules. Edge cases: WO not from a PM rule (no reset), rule deactivated between MR generation and WO closure (rule not found — skip), asset has no lower-level rules (no-op). The backend team must handle these gracefully.

2. **Next-due for reading-based rules needs current meter value.** The resource computation requires the asset's latest confirmed meter reading. If no confirmed readings exist, `progress_percentage` is null and `pm_status` falls back to "ok" (can't determine progress). The UI should show "No readings yet" instead of a status indicator in this case.

3. **Multi-level setup partial failure.** If the backend rejects L2 (e.g., validation error) but accepts L1 and L3, the user gets a partial result. The frontend must clearly show which levels succeeded and which failed, with retry for the failed ones. No transaction rollback — each rule is independent.

4. **Schedule display accuracy.** The day-to-month conversion (180 days → "6 Months") is an approximation. 180 days ≠ exactly 6 calendar months. This is display-only and acceptable for O&G where intervals are guidelines, not exact deadlines.

5. **Cross-plan dependency.** PM Rules moves out of the Admin tab container. The admin plan must be updated to remove PM Rules routes/tabs. Coordinate so neither plan leaves orphan routes.

6. **`usage_reading_type` not in index response.** Prerequisite #2 must also ensure the PmRuleIndexQuery eager-loads `usageReadingType`, not just the `show` endpoint. Without it, the list page can't display the schedule string for reading-based rules.
